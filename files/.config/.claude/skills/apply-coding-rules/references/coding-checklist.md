# Coding Checklist

## 編集前

1. 要求を満たす最小変更は何かを決める
2. どの層まで触る必要があるかを決める
3. 一緒に追加、更新すべき unit test を決める
4. constructor や diコンテナ から注入すべき依存を洗い出す
5. API 契約、config、migration、README、TypeSpec まで追従が必要かを先に確認する

## 実装中

1. 関数、構造体、test 名は説明なしで役割が読める名前にする
2. 1 つの関数に parse、validation、認可、永続化、response 組み立てを詰め込まない
3. 既存の file 粒度と naming を合わせる
4. 依存は inline 生成せず、引数や adapter 越しに受ける
5. コメントは非自明な意図や境界理由だけに使う
6. hack に見える実装になったら、一度止まってもっと素直な配置を探す

## テストと拡張

1. まず最小の working version を作る
2. 段階導入が妥当なときは、最初の成立条件を確認してから scope を広げる
3. 機能変更と同じタイミングで unit test を追加、更新する
4. 2 段階目の改善が必要でも、最初の成立条件と diff を混ぜすぎない

## logging と信頼性

1. silent failure を増やさない
2. log は後から障害調査できる粒度にする
3. token、secret、payload 全体、個人情報の出しすぎを避ける
4. プロジェクトの標準 logger を正規導線にし、独自の logger 受け回しを増やしすぎない

## レビュー観点

- これは本当に最小変更か
- 責務に合う層へ置けているか
- 依存注入の境界が崩れていないか
- 命名だけで役割を追えるか
- コメントは why の補足だけになっているか
- test は意図した振る舞いを証明しているか
- 契約変更に必要な周辺更新が漏れていないか
