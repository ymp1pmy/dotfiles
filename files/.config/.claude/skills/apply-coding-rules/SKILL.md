# Apply Coding Rules

このスキルは、このリポジトリで実装や修正を始める直前に使う。
一般論ではなく、現行の `README.md`、`docs/adr`、既存コードの責務分担に合わせて判断する。

本格的な実装の前に次を読む。

- `references/coding-checklist.md`

## 基本方針

- まず最小の変更で成立させる
- 単純な endpoint でも layering は崩さない
- 既存の配置、命名、責務分担を真似し、独自流儀を足さない
- 依存追加は最後の手段にし、標準ライブラリか既存依存で足りるか先に確認する
- コメントは why / intent の補足に絞って短く書く
- 一時しのぎより根本原因を優先する

## 実装フロー

1. 変更対象がどの責務に属するかを決め、触る層を最小化する
2. 新しい API や機能でも、既存 endpoint と同じ粒度、同じ読み順、同じ命名で実装する
3. 依存は constructor や diコンテナを用いて注入し、下位層で直接生成しない
4. コードと同じタイミングで unit test を追加・更新する
5. 契約や周辺仕様を変えたら、OpenAPI契約、README、config、migration など必要な追従も同じタスクで行う
6. 最後に format、test、diff review までやって完了にする

## 出力契約

- どの層に何を追加、変更したかを説明する
- 追加、更新した test や確認コマンドを伝える
- 未確認事項や段階的な次対応があれば明示する
