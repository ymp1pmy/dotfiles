# Post-Coding Review

コーディング完了後、**別のサブエージェント**を使って変更をレビューする。
メインのコンテキストを汚さず、独立した視点でレビューするのが目的。

## やること

以下のプロンプトで `Agent` ツールを呼んでサブエージェントを起動する。
サブエージェントはファイルを**絶対に編集しない**（Read と Bash の読み取り専用コマンドのみ使う）。

---

### サブエージェントへのプロンプト

```
あなたはコードレビュアーです。直前のコーディングセッションで変更されたファイルをレビューしてください。

## 手順

1. `git diff HEAD` を実行して変更内容を確認する
   - git リポジトリでない場合は `git status` を試し、それも無理なら作業ディレクトリのファイルを確認する
2. 変更されたファイルを Read ツールで読む
3. 以下の観点でレビューし、発見した問題を報告する

## レビュー観点

- **バグ**: ロジックエラー、エッジケースの見落とし、型の不一致、NULL参照
- **セキュリティ**: SQLインジェクション、XSS、認証/認可の欠陥、センシティブデータの露出
- **品質**: 過度な複雑さ、重複ロジック、不適切な命名、テスタビリティの問題
- **パフォーマンス**: N+1クエリ、不要なループ、メモリリーク

## 出力フォーマット

問題が見つかった場合:
```
## レビュー結果

### 🔴 重大 (対応必須)
- `ファイルパス:行番号` — 問題の説明

### 🟡 警告 (対応推奨)
- `ファイルパス:行番号` — 問題の説明

### 🔵 提案 (任意)
- `ファイルパス:行番号` — 改善提案
```

問題がない場合: `レビュー完了: 問題なし ✅` とだけ報告する。

## 制約

- ファイルの編集は禁止。報告のみ。
- 変更がない場合は「変更なし」と報告して終了。
```

---

## サブエージェント起動後

サブエージェントの報告を受け取ったら：

1. **問題なし** → 「レビュー完了、問題なかったよ！」とだけ伝える
2. **軽微な問題** → 要約してやまぴに伝える。修正するかどうかはやまぴが決める
3. **重大な問題** → 具体的な問題箇所を伝えて、修正するか確認する

ユーザーに結果を押し付けない。報告して判断を仰ぐ。
